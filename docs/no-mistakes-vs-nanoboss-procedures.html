Could no-mistakes be rebuilt on nanoboss procedures?

Short answer: yes for the inner workflow engine, no for the outer git-gate shell. A credible rewrite would keep a thin custom gate/hook/worktree wrapper and move the step logic, approvals, and agent orchestration into nanoboss procedures.

Bottom line

Nanoboss is a good fit for the parts of no-mistakes that are really workflow semantics: typed agent calls, fix loops, pause/resume approvals, durable run history, and UI surfaces. It is not a drop-in replacement for the git remote gate, post-receive hook, disposable worktree manager, branch supersession policy, or OS-managed daemon story.

1 · no-mistakes backgrounder

no-mistakes is not just an AI reviewer. It is a git transport gate plus a long-running orchestration service.

Architecture in one view

Working repodeveloper pushes to named remote no-mistakes, not origin
→
Local bare gate repo~/.no-mistakes/repos/<id>.git receives the push
→
post-receive hooknon-blocking notification into the daemon
Daemonbranch locking, run creation, crash recovery, IPC, service integration
→
Disposable worktreesafe place for rebases, edits, tests, commits, pushes
→
Fixed pipelinerebase → review → test → document → lint → push → pr → ci

What makes the product shape unusual

  • Named remote gate: it preserves normal Git semantics; origin is never hijacked.
  • Daemon ownership: pushes stay fast because the hook exits immediately and the daemon does the real work.
  • Ephemeral worktrees: agent edits never touch the developer’s live checkout.
  • SQLite + IPC + Bubble Tea TUI: runs, step states, step rounds, findings, diffs, and logs are durable and attachable.
  • Human checkpoints: approve, fix, skip, abort; auto-fix loops are bounded and recorded.
This means a full rewrite has to respect two very different concerns at once: repository event plumbing and agentic workflow semantics.
The core mental model is “local git proxy + daemonized validation machine,” not “chat app with a review command.”

2 · repository anatomy

The codebase splits cleanly into workflow core, repo plumbing, provider integrations, and UX.

Transport + bootstrap

  • internal/gate: init/eject, bare gate repo, remote wiring
  • internal/git/hook.go: post-receive hook generation
  • internal/daemon/service*.go: launchd/systemd/schtasks lifecycle
  • internal/update: binary replacement + daemon reset

Run engine

  • internal/daemon/manager.go: per-branch locks, cancellation, worktree lifecycle
  • internal/pipeline/executor.go: sequential step execution + approval loop
  • internal/pipeline/steps/*: eight opinionated steps and common helpers
  • internal/db/schema.go: repos, runs, step_results, step_rounds

Agents + providers + UX

  • internal/agent: Claude, Codex, Rovo Dev, OpenCode adapters
  • internal/scm/* + internal/bitbucket: PR/CI hosts
  • internal/tui: step-centric live run view with findings and diff review
  • internal/wizard: onboarding flow to create branch/commit/push

Why the implementation is solid

  • Tests are heavy: 264 Go files, 144 test files, and go test ./... passed in this snapshot.
  • State is explicit: step rounds record findings, selected finding IDs, fix summaries, and durations.
  • Cross-platform concerns are real: service managers, IPC transport, and shell environment reloading are all first-class.

Where the complexity sits

largest workflow hotspots: wizard.go 675 LOC, prsummary.go 647, executor.go 590, manager.go 538 most complexity is product plumbing, not prompt text several subsystems exist only because the product is a git gate, not just a workflow
That split is exactly why nanoboss is attractive: it can absorb the workflow half of the problem, but not the repo-plumbing half by itself.
Important hidden asset: step_rounds already encode a run graph of attempts, selections, and outcomes.

3 · nanoboss capability map

Nanoboss already has most of the primitives needed for the inner half of no-mistakes.

Procedure/runtime primitives

ctx.agent
typed agent calls, fresh/default session modes, named refs, downstream model selection
pause / resume
native continuation state, suggested replies, and form payloads for human checkpoints
ctx.procedures
nested child procedures, so each pipeline step can be its own run with its own typed result
SessionStore
durable runs, refs, traversal, and session metadata instead of bespoke step JSON blobs
dispatch jobs
detached background procedure workers with progress forwarding and recovery helpers
ui.panel / TUI extensions
pipeline cards, status chrome, custom panels, and focus/checkpoint forms without rewriting the whole terminal runtime

Repo-aware proof points already in nanoboss

  • procedures/nanoboss/pre-commit-checks.ts already runs repo validation, auto-fix passes, and continuation-based approval.
  • procedures/linter.ts already does iterative shell-driven repo fixing with nested commits.
  • procedures/simplify2.ts proves that a large resumable harness with forms, panels, local FS, git checks, and agent phases is viable.
  • docs/procedure-engine-package.md makes background dispatch and resumable top-level runs a first-class engine concern.
Procedures are not glorified prompts. They are programmable harnesses with durable state and UI hooks.
Important mismatch: nanoboss natively targets Claude/Codex/Copilot/Gemini via ACP. no-mistakes also supports Rovo Dev and OpenCode, so exact provider parity would require extra adapter work or a product decision.
If you port only the workflow semantics, nanoboss is already much closer than a generic chat harness.

5 · recommended architecture boundary

The cleanest rewrite is a hybrid: thin custom gate wrapper outside, nanoboss procedure package inside.

Proposed flow

git push no-mistakeskeep the existing explicit gate UX
→
thin gate wrapperhook install, branch lock, worktree create/cleanup, active-run index
→
dispatch nanoboss runone nanoboss session per pipeline run, started in the detached worktree
/nm/runtop-level harness procedure
→
child procedures/nm/rebase, /nm/review, /nm/test, /nm/document, /nm/lint, /nm/push, /nm/pr, /nm/ci
→
nanoboss UI + storepanels/forms/cards + durable run graph instead of custom step-event protocol

Why this split is the right one

  • Outer layer stays deterministic and tiny: no prompts, no agent logic, only git/event/process policy.
  • Inner layer becomes reusable: each step is a normal procedure with typed outputs and child-run provenance.
  • History gets richer: prior findings, diffs, and fix attempts can live as refs/run outputs instead of only JSON columns.
  • UI becomes cheaper to evolve: panels + forms are easier to change than a fully bespoke Bubble Tea surface.
  • Reruns become natural: wrapper just dispatches another top-level procedure against a fresh worktree.
I would model each pipeline run as its own nanoboss session and keep a tiny repo/branch index outside nanoboss to answer “what is the current active run for this branch?”
This preserves the product-defining git semantics while collapsing a large amount of custom workflow machinery.

6 · simplifications vs irreducible custom pieces

The win is real, but it is concentrated in executor/agent/UI layers — not in the git gate itself.

Likely simpler after the port

  • Agent backend surface: stop maintaining custom Claude/Codex/Rovo/OpenCode invocation code and lean on ACP-backed nanoboss paths where possible.
  • Approval engine: replace bespoke awaiting_approval / fix_review state transitions with first-class continuations and forms.
  • Provenance model: use nested runs/refs instead of a purpose-built step event protocol plus DB round serialization.
  • TUI customization burden: build pipeline panels/extensions on top of nanoboss instead of owning the full Bubble Tea app stack.
  • Workflow composition: PR summary generation, CI babysitting, and repo validation become ordinary procedures instead of tightly coupled step types.

Still custom, or potentially harder

  • Gate init/eject + remote semantics: nanoboss has no opinionated git-remote interception layer.
  • Managed daemon / service / update lifecycle: if you still want hands-off background execution, this remains your responsibility.
  • Branch-oriented querying: nanoboss is session/run oriented, not repo+branch database oriented.
  • Provider host ports: GitHub/GitLab/Bitbucket CI/PR code has to move or be wrapped.
  • Exact UX parity: the current no-mistakes step-box UI is a better at-a-glance pipeline monitor than a default transcript UI.
I would plan to delete most of the custom executor and agent plumbing — but not the thin gate wrapper, and not every bit of repo-local state indexing.
In practice: nanoboss can replace the orchestration core, not the product shell.

7 · verdict

My recommendation

If the goal is to make no-mistakes less bespoke in its workflow engine, yes — nanoboss procedures are a credible target. If the goal is to replace the whole product with “just procedures,” no — the git-gate, worktree, branch-coordination, and service-management shell still has to exist.

Could it be done?

Yes, with a hybrid architecture.
  • Port the fixed pipeline to a procedure package.
  • Use nested child procedures for each step.
  • Use nanoboss continuations for approvals/fixes.

Could it be done with procedures alone?

No.
  • Git hook intake is external.
  • Detached worktree lifecycle is external.
  • Per-branch cancellation policy is external.

Would I recommend it?

Yes, if you accept some product reshaping.
  • Keep the explicit gate UX.
  • Shrink the custom core around it.
  • Expect some TUI/provider migration work.

Suggested migration order

  1. Implement a repo-local .nanoboss/procedures/no-mistakes/* package for the eight steps and the top-level /nm/run harness.
  2. Build a tiny wrapper that receives push events, creates the worktree, and dispatches the top-level nanoboss run.
  3. Add a nanoboss TUI extension/panel renderer for a step-ladder pipeline view and approval actions.
  4. Port PR/CI provider hosts and decide whether to keep a tiny repo index or continue using SQLite for branch lookup.
Shortest summary: rewrite the engine on nanoboss; do not pretend the git gate disappears.
