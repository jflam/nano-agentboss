Barnum is a workflow compiler.
nanoboss procedures are a durable command runtime.

They live in the same agent-orchestration neighborhood, but they optimize different layers. Barnum turns orchestration into a typed AST and executes it as a Rust-managed state machine. nanoboss keeps procedures as ordinary TypeScript modules and surrounds them with session state, UI events, durable run storage, continuation support, and downstream-agent policy.

Executive summary

The short version of what matters architecturally.

Barnum’s core idea

Author a workflow as a typed DSL, serialize it, compile it to a compact runtime plan, and let a Rust event loop drive the state machine.

  • Workflow structure is explicit and machine-checkable.
  • Concurrency, branching, looping, racing, and retries are engine primitives.
  • Every TypeScript handler runs in an isolated subprocess.
  • Zod contracts become both compile-time types and runtime validators.

nanoboss’s core idea

Author a procedure as normal TypeScript, then let the procedure engine supply agent calls, nested procedures, session policy, run persistence, UI, and pause/resume semantics.

  • The workflow is mostly the host language’s control flow.
  • Durability is first-class: runs, refs, prompt images, metadata, continuations.
  • Agent calls are bounded through ctx.agent.run(...).
  • Human-in-the-loop experiences are built into the model.

Bottom line

  • Barnum is stronger when the primary problem is precise orchestration semantics.
  • nanoboss is stronger when the primary problem is interactive agent tooling inside a persistent operator session.
  • They are more complementary than competitive.
  • The most interesting future shape would be: Barnum-like subworkflow semantics inside nanoboss’s durable session/runtime shell.

Barnum deep dive

Barnum is not just “TypeScript orchestration code.” It is a compiled workflow system with a real runtime model.

1) Authoring model

TypeScript DSL
TypedAction<In, Out> values compose with pipe, all, forEach, branch, loop, race, and friends.
↓
Serializable AST
Non-enumerable methods and phantom types disappear under JSON.stringify(), leaving a clean workflow tree.
↓
Rust engine input
The Rust side receives a declarative action graph rather than arbitrary user code.
  • createHandler(...) captures module path + export name and embeds JSON Schema for input/output validators.
  • The AST is closed over nine action kinds: Invoke, Chain, All, ForEach, Branch, ResumeHandle, ResumePerform, RestartHandle, RestartPerform.
  • The user-facing API feels fluent, but the runtime sees a strict algebra of nodes.

2) Compilation and execution

  • The Rust compiler flattens the tree into a linear FlatConfig, replacing pointers with indexed cross-references.
  • Identical handlers are interned so schemas and handler identities are compiled once per unique handler, not once per call site.
  • The runtime uses a frame arena + pending effect queue. advance() expands structure; complete() delivers results upward.
  • The event loop drains pending effects before waiting for external completions, which makes restart/race semantics precise.
  • Stale completions are dropped via liveness checks after restarts or races tear down parts of the frame tree.

3) Isolation, effects, and validation

  • Isolation: TypeScript handlers run in isolated Node subprocesses via worker.ts; builtins run in-process.
  • Effects: loop, tryCatch, earlyReturn, race, and withTimeout are compiled onto algebraic-effect-style restart/resume handlers.
  • Validation: Zod → JSON Schema at definition time, then Rust Validator objects validate handler input pre-dispatch and output post-completion.
  • Result: Barnum minimizes hidden state and keeps each step narrow, explicit, and machine-governed.

Representative Barnum shape

runPipeline( listFiles.forEach( pipe( refactor, evaluate, loop((recur, done) => pipe(typeCheck, classifyErrors).branch({ HasErrors: pipe(forEach(fix).drop(), recur), Clean: done, }) ), commit, createPR, ) ), );

The important thing here is not syntax sugar. The whole control-flow structure is present in the runtime model.

How nanoboss procedures work

nanoboss procedures are runtime-integrated command modules, not compiled workflow graphs.

1) Authoring and discovery

Procedure module
A default export satisfies Procedure: name, description, execute(...), optional resume(...), optional cancel(...).
↓
Registry layer
Builtins are lazily registered; disk procedures are discovered recursively by scanning .ts files for static metadata and a default export.
↓
Execution entry
executeProcedure(...) creates the top-level run and builds a concrete CommandContextImpl.
  • Procedures stay as ordinary TypeScript/Bun code — no AST compilation step.
  • Control flow is mostly explicit await, if, loops, and helper functions in the host language.
  • The procedure engine provides the runtime shell, not a second workflow language.

2) The procedure context is the real platform

  • ctx.agent: the only supported route for downstream agent calls; supports typed JSON descriptors, refs, model selection, and fresh/default session modes.
  • ctx.procedures: nested procedure calls with inherit, default, or fresh session binding.
  • ctx.state: read-only durable run/ref inspection over SessionStore.
  • ctx.session: live default-agent selection and token snapshots/usage.
  • ctx.ui: text, status, cards, and structured procedure panels.

3) Persistence and continuation are first-class

  • Every top-level procedure, child procedure, and agent call becomes a durable run record.
  • Run output can be addressed through stable refs instead of being held only in memory.
  • Prompt images and session metadata are persisted alongside the run graph.
  • A procedure can return pause: { question, state, ... }; later the engine resumes through the procedure’s resume(...) hook.
  • Async/background work is supported with detached procedure-dispatch workers, job JSON files, progress JSONL, cancellation files, and recovery helpers.

Representative nanoboss shapes

// research.ts const brief = expectData(await ctx.agent.run( buildResearchBriefPrompt(prompt), ResearchBriefType, { session: "default", stream: false }, )); const report = expectData(await ctx.agent.run( buildResearchExecutionPrompt(prompt), ResearchResultType, { refs: { brief: briefRef }, stream: false }, )); return { data, display, summary }; // simplify.ts return { pause: { question, state, suggestedReplies, }, };

The nanoboss style is imperative orchestration wrapped by durable runtime services.

Side-by-side comparison

The sharpest distinction: Barnum compiles orchestration semantics; nanoboss operationalizes command execution inside a long-lived session product.

Intermediate representation Explicit serializable AST, then flattened FlatConfig. No workflow IR; the host language remains the control-flow representation.
Runtime substrate Rust event loop + frame arena + pending-effect queue. JavaScript/Bun async execution + procedure engine services + durable store.
Concurrency model First-class via All, ForEach, race. Mostly manual via host code; background dispatch exists at whole-procedure granularity.

Control-flow expressiveness Loops, restarts, races, and effect handling are engine-native. Mostly imperative JS patterns plus explicit pause/resume at the procedure boundary.

The most important conceptual difference

These systems put the “smartness” in different places.

Barnum pushes intelligence into the engine

  • The runtime understands the workflow’s structure.
  • The scheduler knows what is parallel, what is restartable, what is in-flight, and what becomes stale.
  • The language and runtime cooperate to reduce hidden context and hidden control flow.
  • It is closer to a workflow VM than to a command plugin system.

nanoboss pushes intelligence into the runtime shell

  • The engine does not own a compiled procedure graph; it owns execution boundaries, persistence, and product behavior.
  • The important abstractions are session state, child runs, refs, continuation state, UI emission, and agent policy.
  • Procedures are closer to commands with powerful services than to declarative state machines.
  • It is closer to an application runtime for agent tools than to a dedicated workflow language.

Cross-pollination ideas

If you wanted these codebases to learn from each other, these are the highest-leverage imports.

Ideas nanoboss could borrow from Barnum

  • Declarative subworkflows inside procedures. A Barnum-like mini DSL for fan-out, branch, race, and retry would remove a lot of imperative boilerplate from complex procedures like simplify2.
  • Engine-visible topology. If nanoboss could optionally compile subgraphs, the runtime could visualize, analyze, and debug orchestration plans before running them.
  • Boundary validation between substeps. Barnum’s “validate input before dispatch, validate output before delivery” model is very strong.
  • Step isolation as a policy tool. Barnum’s subprocess boundary is a practical way to reduce accidental state bleed across agentic steps.

Ideas Barnum could borrow from nanoboss

  • Durable run graph + refs. Barnum currently feels ephemeral compared with nanoboss’s session store.
  • Pause/resume continuations. nanoboss has a concrete model for human checkpoints and resumable commands.
  • Operator-facing UI events. nanoboss procedures can surface status, panels, cards, and async job progress cleanly.
  • Persistent default agent sessions. Barnum intentionally isolates steps, but some workflows benefit from a reusable conversational memory channel.

My synthesis

Barnum solves the “how do I make the orchestration itself precise?” problem. nanoboss solves the “how do I make agent commands durable, resumable, inspectable, and productized?” problem. A hybrid would be compelling: Barnum-like compiled orchestration inside nanoboss’s durable session/runtime shell.
